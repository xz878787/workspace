# 前端面试高频题：扁平列表转树状结构，两种经典写法一次搞懂

## 开篇

在前端开发中，**列表转树状结构**是一个出镜率极高的场景——省市区三级联动、后台管理系统的多级菜单、组织架构树、文件目录树……它们的后端数据几乎都是同一种形态：

```js
// 后端返回的扁平数据
[
  { id: 1, name: '数学教研组', parentId: 0 },
  { id: 2, name: '语文教研组', parentId: 0 },
  { id: 3, name: '高一年级', parentId: 1 },
  { id: 4, name: '高二年级', parentId: 1 },
  { id: 5, name: '高一(1)班', parentId: 3 },
]
```

数据库里存的就是这个结构，`SELECT * FROM department` 出来就是这样一张平表。前端需要把它变成嵌套的 `children` 树，才能交给级联选择器或树组件渲染。

这篇文章会把思路拆开讲透，并给出两种经典写法。

## 为什么用 parentId

扁平列表的核心关联字段是 **parentId**——每一项通过 `parentId` 指向自己的父节点。`parentId: 0` 表示这是一级节点，没有父级。

这种设计的好处是：
- **存储简单**：数据表和业务逻辑解耦，增删改都是单行操作
- **递归自由**：理论上可以无限层级，不需要预留嵌套层数
- **前后端通用**：同一份数据，后端做权限过滤、前端做树形渲染，各取所需

你写的 `WHERE parent_id = ?` 查一层，前端转成树再渲染。

## 核心思路：两次遍历

无论用哪种写法，核心理念都一样——**用哈希表建立 id → 节点的映射，然后挂载父子关系**。

```
第一步：遍历列表，为每个节点创建带 children 空数组的副本，存入 map
第二步：再次遍历，通过 parentId 从 map 中取出父亲，把当前节点 push 进父亲的 children
第三步：parentId 为 0（或查不到父亲）的节点，就是根节点，直接进 tree 数组
```

时间复杂度 O(n)，空间复杂度 O(n)。两次遍历、没有嵌套循环，十万条数据毫秒级出结果。

## 写法一：Map + forEach

```js
const flatList = [
  { id: 1, name: '一级菜单A', parentId: 0 },
  { id: 2, name: '一级菜单B', parentId: 0 },
  { id: 3, name: '二级A-1', parentId: 1 },
  { id: 4, name: '三级A-1-1', parentId: 3 },
  { id: 5, name: '二级B-1', parentId: 2 },
];

function listToTree(list) {
  const map = new Map();
  const tree = [];

  // 第一遍：建立 id → 节点 的映射
  list.forEach(item => {
    map.set(item.id, {
      ...item,
      children: [],
    });
  });

  // 第二遍：挂载父子关系
  list.forEach(item => {
    const current = map.get(item.id);
    const parent = map.get(item.parentId);
    if (parent) {
      parent.children.push(current);
    } else {
      tree.push(current);
    }
  });

  return tree;
}
```

`Map` 在这里的优势是语义清晰——`get` / `set` 一看就懂，而且 key 可以是任意类型。如果你的 `id` 是数字和字符串混用（后端没统一），`Map` 不会做隐式转换，比普通对象更安全。

## 写法二：reduce + 普通对象

```js
function listToTree(list) {
  // 第一遍：reduce 建映射表
  const nodeMap = list.reduce((map, item) => {
    map[item.id] = { ...item, children: [] };
    return map;
  }, {});

  // 第二遍：reduce 构建树
  return list.reduce((tree, item) => {
    const cur = nodeMap[item.id];
    const parent = nodeMap[item.parentId];
    if (parent) {
      parent.children.push(cur);
    } else {
      tree.push(cur);
    }
    return tree;
  }, []);
}
```

这种写法更"函数式"——`reduce` 一气呵成，没有外部变量。如果你团队偏好链式调用和不可变风格，这个版本更合群。

两种写法本质完全相同，挑你顺眼的就行。面试官一般两种都认。

## 进阶：你可能会遇到的变体

**1. 数据没有 parentId，只有一个 pid 字段**

```js
// 加一行映射即可
const normalized = list.map(item => ({
  ...item,
  parentId: item.pid, // 兼容老接口
}));
```

**2. 顶层节点的 parentId 不是 0，是 null / undefined / -1**

```js
const isRoot = parentId => parentId == null || parentId === 0 || parentId === -1;
```

判断 `!parent` 本身就覆盖了 `null` 和 `undefined` 的情况。

**3. 需要保留原对象引用（不展开 `...item`）**

```js
item.children = [];
map.set(item.id, item);
```

直接污染原数据，省一次浅拷贝。取决于你的业务能不能接受副作用。

**4. 转完树之后还要支持"拍平回去"**

```js
function treeToList(tree) {
  const result = [];
  const walk = nodes => {
    nodes.forEach(node => {
      const { children, ...rest } = node;
      result.push(rest);
      if (children.length) walk(children);
    });
  };
  walk(tree);
  return result;
}
```

树转列表是递归遍历，列表转树是哈希映射，两个方向都要会。

## 总结

| 维度 | 写法一 (Map) | 写法二 (reduce) |
|------|-------------|-----------------|
| 映射结构 | `new Map()` | 普通对象 `{}` |
| 代码风格 | 命令式 forEach | 函数式 reduce |
| 时间/空间 | O(n) / O(n) | O(n) / O(n) |
| key 类型 | 任意类型 | 字符串/Symbol |

核心就三句话：**建映射 → 找父亲 → 挂孩子**。理解了这六个字，这道题就算真正掌握了。
