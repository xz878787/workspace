# Workflow 与 Agent：AI 工程的两条路径

> 一条是高速公路，目的地明确，效率极高；另一条是越野探索，没有既定路线，却可能发现全新的风景。

---

## 一、引言：两个被混淆的概念

在 AI 应用开发的日常讨论中，"Workflow" 和 "Agent" 经常被混为一谈。不管是用 LangChain 搭建的链式调用，还是在 Coze 上拖拽的可视化流程，大家习惯于说"我在做一个 AI 智能体"。

但严格来说，这两者走的完全是两条路。

如果你用 LangChain 的 `pipe` 方法把 prompt、模型调用、输出解析器串成一条链，这就是 Workflow。而如果你给一个模型配备了工具、记忆、规划能力，让它自己去决定每一步做什么，这就是 Agent。

区分这两者，不是为了咬文嚼字，而是因为在工程实践中，**选错了范式，要么失控，要么白费**。

---

## 二、AI Workflow：确定性的流水线

### 2.1 本质定义

Workflow 是一套**预定义的执行流程**。把它想象成一条工业生产的流水线：每个工位（节点）有明确的输入和输出，每个工人（节点逻辑）只做自己分内的事。上一个工位完成了，物料传送到下一个工位，整个过程中不会有人突然"灵机一动"去做点别的。

从技术角度看，Workflow 就是一个**编排好的工作流引擎**（Orchestration Engine）。它通过预设的节点和边，将多个处理步骤串联或并联起来，形成一条完整的任务处理链路。

### 2.2 形象比喻：传送带

用 LangChain 来打个比方：

```python
creativeChain = storyPrompt.pipe(creativeModel).pipe(outputParser)
creativeChain.invoke("恭喜拿下offer")
```

这里的 `pipe` 就是传送带。一份输入从 `storyPrompt` 这头进去，经过 `creativeModel` 加工，再被 `outputParser` 整理，最后输出。每一步走什么路径、调用什么模型、做什么处理，都是**提前写死在代码里的**。

不管你调用 `creativeChain.invoke` 多少次，它走的路永远一样——就像你工厂流水线上一瓶一瓶的可乐，第 1 瓶和第 10 万瓶，工序毫无区别。

### 2.3 一个招聘场景的例子

假设你要用 AI 处理一批简历，典型的 Workflow 长这样：

1. **解析 PDF**：把简历文件转成可读文本
2. **信息抽取**：从文本中提取技能描述、工作经历、教育背景等结构化字段
3. **岗位匹配**：将抽取出的信息与岗位需求做 RAG（检索增强生成），计算匹配度
4. **生成报告**：输出一份包含评分和建议的筛选报告

这四个步骤，顺序固定，逻辑固定。第 1 步永远在解析 PDF，不会突然决定"我觉得这个候选人很有意思，我先去搜一下他的 GitHub"。这就是 Workflow 的确定性——**它不会给你惊喜，也不会给你惊吓**。

### 2.4 适合 Workflow 的场景

从产业角度看，Workflow 特别适合**流程化、可重复、对稳定性要求高**的任务：

- **审核流程**：内容审核、合规审查，每一步的规则必须可解释、可追溯
- **数据清洗**：ETL 管道，从抽取到转换到加载，不允许跳过任何一个步骤
- **客服 AI**：标准问题走固定应答流程，FAQ 匹配 → 知识库检索 → 转人工坐席
- **报表生成**：定期从多个数据源拉取数据，按固定模板输出

### 2.5 技术栈特征

Workflow 的技术栈更依赖**规则和编排**：

- **可视化编排工具**：Coze、Dify、n8n 等低代码平台通过拖拽节点来构建流程
- **流程引擎**：LangChain 的 Chain/LCEL、Temporal、Airflow 等负责调度执行
- **状态管理**：每一步的状态、中间结果需要显式管理和传递
- **错误处理**：异常分支需要预先定义，比如"解析失败则标记为异常并跳过"

## 三、Agent：不确定性的探索者

### 3.1 本质定义

Agent 和 Workflow 的逻辑完全不一样。它**不走在预设的轨道上**，而是站在一个开放的空间里，自己去探索怎么完成任务。

一个 Agent 从技术角度需要具备**三个核心能力**：

**第一，感知环境（Perception）**

Agent 需要通过输入（文本、图像、数据）来理解当前的任务是什么、有哪些工具可以调用、目前进展到哪一步了。这种感知不是被动接收，而是主动分析——"我现在手里有什么，我还缺什么，我应该先做什么？"

**第二，规划路径（Planning）**

不同于 Workflow 的预定义节点链，Agent 的路径是**动态生成的**。它根据当前感知到的信息，实时拆解任务、决定步骤顺序。同一个任务，执行三次可能走三条不同的路径。

**第三，执行行动（Action）**

Agent 调用工具（搜索、计算、API 调用、代码执行等）来完成子任务，并根据执行结果的反馈**持续调整**后续策略。这不是"执行-结束"的单向流程，而是"执行-观察-调整-再执行"的循环——也就是经典的 ReAct（Reasoning + Acting）模式。

### 3.2 形象比喻：司机 vs 高速公路

这个比喻可能最好理解：

**Workflow 是一条高速公路**——路线清晰、固定，效率很高，但只能到达既定的终点。你上了这条高速，沿途的出口是预设好的，你不能临时决定拐进一条乡间小路。

**Agent 则是一个司机**——他知道目的地在哪里，但随时可以变通。看到前方堵车（感知环境），他会绕路；听说某条路风景更好（新信息），他会改道；甚至当你突然说"不去杭州了，去苏州吧"（目标改变），他也能立刻调整导航。

这个司机的"自由度"就是 Agent 的核心价值——**自主性和适应性**。

### 3.3 一个旅行规划的例子

你对 Agent 说："帮我规划一趟去日本的旅行。"

Agent 的思考过程可能是这样的：

1. **感知环境**：先去搜索东京和大阪的机票价格，再看几个城市的酒店报价
2. **发现新变量**：发现京都正好是樱花季，酒店已经快订满了，价格飙升
3. **动态调整**：查询签证要求、天气预测、当地交通状况
4. **权衡决策**：结合你之前提到过的偏好（喜欢安静、预算中等），最终推荐名古屋作为备选，或者调整出行日期避开高峰期

整个过程中，Agent 走的路径可能和你的预期完全不同。它可能发现了你不知道的信息（樱花季溢价），提出了你没考虑过的方案（名古屋替代京都），然后自己权衡出了一个最优解。

这就是 Agent 和 Workflow 的关键区别——它不会被一个固定的流程限制，而是**像一个思考中的个体**。

### 3.4 适合 Agent 的场景

Agent 更适合**动态、复杂、需要灵活应变的场景**：

- **复杂搜索与研究**：遍历多个信息源，根据中间发现不断调整搜索方向
- **策略规划**：在多个约束条件下动态寻找最优解，如供应链优化、投资组合配置
- **代码开发助手**：理解需求 → 设计方案 → 编写代码 → 调试修复，路径无法提前确定
- **多轮谈判与对话**：根据对方的反馈实时调整策略和话术

### 3.5 技术栈特征

Agent 的技术栈更依赖**大模型的核心能力**：

- **推理能力**：模型本身需要有足够强的逻辑推理来做规划和决策
- **工具调用（Function Calling / Tool Use）**：模型能自主选择调用什么工具、以什么参数调用
- **记忆系统**：短期记忆（上下文）和长期记忆（向量数据库存储的历史经验）
- **反思机制**：执行后对结果进行评估，必要时回溯和重新规划

## 四、核心对比：Workflow vs Agent

| 维度 | Workflow | Agent |
|------|----------|-------|
| **执行方式** | 预定义、确定性 | 动态探索、不确定性 |
| **路径** | 固定节点链，每次相同 | 实时生成，每次可能不同 |
| **决策权** | 在开发者手中 | 在模型手中 |
| **可控性** | 高，每一步可预测 | 低，结果和路径都不可完全预期 |
| **创造力** | 几乎没有 | 强，能发现新方案 |
| **适应性** | 差，遇到未预设的情况会中断 | 强，能根据环境变化自我调整 |
| **效率** | 高，没有"思考"开销 | 相对低，需要多轮推理和工具调用 |
| **成本** | 低，调用次数固定 | 高且不稳定，调用次数不确定 |
| **调试难度** | 低，路径可复现 | 高，问题难以复现 |
| **典型技术** | 规则引擎、DAG 编排、LangChain Chain | ReAct、Function Calling、记忆系统 |
| **典型平台** | Coze 工作流、Dify 工作流 | AutoGPT、Cline、Manus |
| **适合任务** | 流程化、重复性任务 | 开放式、创造性任务 |

## 五、本质之差：确定性与不确定性的博弈

用一句话概括最根本的区别：

> **Workflow 追求的是确定性的执行，Agent 追求的是不确定性的探索。**

Workflow 没有创造力，但它忠实地完成给定的流程——这在很多企业场景中恰恰是**优点**。你不会希望一个财务报表生成系统在某一天突然"灵机一动"换了一种计算方式。

Agent 有创造力，但这种创造力的代价是**不可控和不可预期**。你可能让它规划旅行，它给出的路线确实很妙；但也可能在某个环节它产生幻觉，调用了错误的工具，把一趟行程搞得面目全非。

在工程上，这对应着两种不同的信任模型：

- 对 Workflow，你信任的是**流程设计**——只要每个节点逻辑正确，整体结果就正确
- 对 Agent，你信任的是**模型能力**——你需要模型足够聪明，能在开放空间里做出正确决策

## 六、未来方向：融合而非对立

在实际的 AI 工程中，最高效的系统往往不是纯 Workflow 也不是纯 Agent，而是**两者的有机结合**。

可以这样理解这层关系：

### 底层是 Workflow，提供骨架

Workflow 负责**稳定性和可控性**。它是系统的骨架，确保核心流程在自己掌控之中。比如数据流入的预处理、合规检查、结果输出的格式校验——这些步骤必须确定，不容有任何意外。

### 上层是 Agent，提供大脑

Agent 负责**灵活性和智能性**。它是嵌在 Workflow 骨架中的大脑，在需要"动脑子"的环节接管决策。比如在客服系统中，标准问题走 Workflow 的固定应答流，但当用户的问题超出了 FAQ 范围，Agent 就会接管，动态搜索知识库、调用工具、与用户多轮对话。

### 一句话总结

> **Workflow 是骨架，Agent 是 Workflow 的大脑。**
>
> Workflow 让系统高效、可靠、可审计；Agent 让系统聪明、灵活、有智慧。两者的融合，才能构建出既稳定又智能的 AI 应用。

---

## 七、结语

回过头看，Workflow 和 Agent 的对立是表面的，互补才是本质。

在动手构建一个 AI 应用之前，先问自己一个问题：**这个任务的核心，是需要稳定性还是灵活性？**

- 如果答案是稳定性 → Workflow 是你的首选
- 如果答案是灵活性 → Agent 能让系统拥有真正的"智能"
- 如果答案是"两者都需要" → 那么恭喜你，你走在正确的路上——用 Workflow 搭骨架，用 Agent 装大脑

未来那些真正产生价值的 AI 工程，大概率不会纠结于"选 Workflow 还是选 Agent"的二元问题，而是熟练地在系统中为不同环节匹配合适的范式。流水线负责效率，智能体负责智慧，这才是 AI 落地的正确姿势。
